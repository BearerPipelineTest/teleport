Jenkins is a open source automation server that is frequently used to build
Continuous Integration and Continuous Delivery (CI/CD) pipelines.

In this guide, we will demonstrate how to migrate Jenkins workloads to
Machine ID with minimal changes to your pipelines.

## Architecture

Before we begin, it should be noted that Jenkins is a tool that is notoriously
difficult to secure. Machine ID is one part of securing your infrastructure,
but it alone is not sufficient.

The simplest Jenkins deployments have the controller and agents run on the
same host. This deployment model is simple to get started with, however any
compromise of the `jenkins` user within a single pipeline can lead to the
compromise of your entire CI/CD infrastructure.

A slightly more complex, but more secure, deployment is running your Jenkins
controllers and agents on different hosts and pinning workloads to specific
agents. This is a improvement over the simple deployment because you can limit
the blast radius of the compromise of a single pipeline to a subset of your
CI/CD infrastructure instead of all of your infrastructure.

We strongly encourage the use of the second deployment model whenever possible,
with ephemeral hosts when possible. When using Machine ID with this model,
create and run Machine ID bots per-pipeline. This will allow you to give each
pipeline the minimal set of server access, reduce the blast radius if one
pipeline is compromised, and allow you to remotely audit and lock pipelines if
you detect malicious behavior.

## Prerequisites

You will need the following tools to use Teleport with Jenkins.

- [Teleport Enterprise](../../enterprise/introduction.mdx) or [Teleport Cloud](../../cloud/introduction.mdx) >= 9.0.0 
- `ssh` OpenSSH tool
- `Jenkins`

In addition, if you already have not done so, follow the
[Machine ID Getting Started Guide](../../docs/pages/machine-id/getting-started.mdx)
to create a bot user and start Machine ID.

## Step 1/2 Start Machine ID

Machine ID allows you to use Linux ACLs to control access to certificates on
disk. You will use this to limit the access Jenkins has to the short-lived
certificates Machine ID issues.

In the example that follows, we'll work with the assumption you will be running
Machine ID as the Linux user `teleport` and Jenkins as the Linux user
`jenkins`. Create and initialize the directories you will need using the
`tbot init` command.

<Details scope={["cloud"]} scopeOnly={true}>
```code
$ tbot init \
    --destination-dir=/opt/machine-id \
    --bot-user=teleport \
    --owner=teleport:teleport \
    --reader-user=jenkins \
    --auth-server=example.teleport.sh:443
```
</Details>
<Details scope={["oss","enterprise"]} scopeOnly={true}>
```code
$ tbot init \
    --destination-dir=/opt/machine-id \
    --bot-user=teleport \
    --owner=teleport:teleport \
    --reader-user=jenkins \
    --auth-server=auth.example.com:3025
```
</Details>

Next, create a systemd.unit file at `/etc/systemd/system/machine-id.service`.
This will allow systemd to start and run Machine ID in the background.

<Details scope={["cloud"]} scopeOnly={true}>
```code
[Unit]
Description=Teleport Machine ID Service
After=network.target

[Service]
Type=simple
User=teleport
Group=teleport
Restart=on-failure
ExecStart=/usr/local/bin/tbot start --destination-dir=/opt/machine-id --token=00000000000000000000000000000000 --ca-pin=sha256:1111111111111111111111111111111111111111111111111111111111111111 --auth-server=example.teleport.sh:443
ExecReload=/bin/kill -HUP $MAINPID
PIDFile=/run/machineid.pid
LimitNOFILE=8192

[Install]
WantedBy=multi-user.target
```
</Details>
<Details scope={["oss","enterprise"]} scopeOnly={true}>
```code
[Unit]
Description=Teleport Machine ID Service
After=network.target

[Service]
Type=simple
User=teleport
Group=teleport
Restart=on-failure
ExecStart=/usr/local/bin/tbot start --destination-dir=/opt/machine-id --token=00000000000000000000000000000000 --ca-pin=sha256:1111111111111111111111111111111111111111111111111111111111111111 --auth-server=auth.example.com:3025
ExecReload=/bin/kill -HUP $MAINPID
PIDFile=/run/machineid.pid
LimitNOFILE=8192

[Install]
WantedBy=multi-user.target
```
</Details>

Finally, to start Machine ID, run the following commands.

```code
$ sudo systemctl daemon-reload 
$ sudo systemctl start machineid
$ sudo systemctl status machine-id
```

## Step 2/2. Update Jenkins pipelines

Using Machine ID within a Jenkins pipeline is now a one line change. Replace any invocation of `ssh` with the following.

```
steps {
    sh "ssh -F /opt/machine-id/ssh_config root@node-name.example.com ..."
}
```
